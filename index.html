Checkpoint strategy

Rotate providers while keeping completion friction low.

Checkpoint 1

  • Shortlink provider A
  • Optional captcha
  • Token TTL: 2 minutes

Checkpoint 2

  • Ad view or offerwall
  • Adblock fallback path
  • Server-side validation

Checkpoint 3

  • Captcha challenge
  • Cooldown enforcement
  • One-time proof token

Anti-bypass plan

Protect keys from replay, spoofing, and automated completion.

Session binding

Every request requires a server-issued session token and fingerprint match.

Signed checkpoint proofs

Use HMAC/JWT tokens with short TTL and one-time redemption.

Rate limits & cooldowns

Enforce IP + fingerprint throttles and key request cooldown windows.

Adblock fallback

If ad scripts are blocked, reroute to captcha or alternate provider.

Endpoint blueprint

Define the minimum API surface to keep keys secure and auditable.

Session

  • POST /api/session/start
  • GET /api/session/status
  • POST /api/session/refresh

Checkpoints

  • POST /api/checkpoint/complete
  • GET /api/checkpoint/next
  • POST /api/checkpoint/verify

Keys

  • GET /api/key
  • POST /api/key/validate
  • POST /api/key/revoke

Data model essentials

Checkpoint

Key

value, session_id, issued_at, expires_at, redeemed_at
